## 〇、一句话总判（先给结论）

**不是"推倒重写"那种大修, 而是补一个被结构性漏掉的环节。** 当前可读性问题里, **句内局部的(换词/寒暄)prompt 渐近能治且大多已治到位; 全局涌现的(密度起伏/详略/节奏/章内重复)有硬天花板, prompt 在"段级"根本够不到"章级"决策面; 结构硬指令自身制造的(perSec=900填满 + 7块硬注入逼cram)prompt 越调越自相矛盾。** 后两类是当前 5 个问题里的 4 个, 它们的共同病根是**生成流水线缺"全章回看修订"这一环 + 把"信息量"硬绑在"字数"上**。最高 ROI 的最小大修 = **章级自修订/编辑 pass(候选 A)**——它一招同时压密度/啰嗦/节奏/章内重复, 因为它补的是模型单遍向前写时**物理上没有的"整章已写成什么样"的视野**。但务实路线是: **先把可读性体检的 prompt 补丁打满(M1/M2/M3 已基本到顶)+ 观测 → 不够再上 A 这个"最小大修", 而非一上来推倒。** 现在的实测已经证明"不够"了(见一·铁证), 所以**触发条件已满足, A 可以上**, 但必须带确定性护栏。

## 1.2 分界线(理论根据): 按"单遍生成时模型能否自控"三分

这是判定"prompt 治标 vs 架构病根"的**唯一可操作判据**:

| 类 | 定义 | 例 | 能否 prompt 治 |
|---|---|---|---|
| **A 类·局部句内** | 写每一句时当下即可判违规, 不需全局视野 | 「别用仿佛/似乎」「对白别寒暄」 | ✅ **渐近能治好**, 加一条措辞违规率单调降、会收敛 |
| **B 类·全局涌现** | 只在整章尺度才有定义, 需"写完回看"才暴露 | 密度疏密相间 / 详略分配 / 节奏快慢 / 章内重复 | ❌ **有硬天花板**。模型单遍自回归向前写时**没有"整章已写成什么样"的表征**, 你要它一边写第3段一边记得前2段密度并补偿=对单遍生成提了个全局约束, 物理上按不住 |
| **C 类·结构自造** | 结构层硬指令直接制造问题, 补丁与结构对抗 | perSec=900填满逼堆料 / 7块硬注入逼cram / persona「须带料登场」对抗留白 | ❌ **永远治不动**, 因为病根是另一条硬指令在反向施压, 加再多"克制"措辞都被结构约束赢 |

**关键证据(B类物理上按不住的源码自证)**: `warm-fitness.ts:141` 注释自己记了血泪教训——物象密度三种测法(总命中/去重种类/重复度)**都区分不开**温情好/坏章, 因为「一两件物反复回扣」(想要的)和「每段堆新物」(不想要的)**在局部无法区分**, 密度是**涌现量、只在整章尺度才有定义**。这正是 B 类撞墙的源码级铁证, 也解释了为何 fitness 信号抓不住单章啰嗦(那条路已被实证堵死, 不要再试)。

**关键证据(C类自相矛盾的源码自证)**: `longrun.ts:203` 同一段 secPrompt 里**既写**「宁短勿堆…不为凑字数往段里堆物象」**又写**「约 900 字」。这是**自相矛盾的指令**, 模型在二者间漂 → dlg/1k 隔章跳 5.6↔11.9 的来源就是这里。

## 1.3 五个可读性问题的归因(哪些到天花板必须动结构)

| 问题 | 类 | prompt 可治部分(多已治) | 必须动架构部分 |
|---|---|---|---|
| **信息密度过高** H-1 | B+C | 换新词造成的密度(fP/pP 已夹紧✅) | perSec=900 逼出的"凑字重描"(C类根) |
| **详略不当/拖沓** H-2 | **纯C/结构** | — | **全部**: 固定 4段×900 把详略写死, 不由内容定 |
| **节奏单一(无快慢段)** H-3 | **纯结构** | — | **全部**: 段均长 900 + 段同采样基因, 章内不可能有起伏(卷级rhythm旋钮管不到段间) |
| **跨章重复** M-1(跨章) | 已治✅ | sceneShift+rollSummary剔静物+W_var跨章Jaccard(3世界var≈9.7近满分=已生效) | — |
| **章内重复**(同物描4遍) M-1(章内) | — | **结构盲区**: W_var 是章间 pairwise Jaccard, **看不见单章内**同物被描4次(章内重复在同一set里被去重了)→进化层结构性无法对它施压 |
| **单章质量改不掉** L-1 | — | 缺章级环节(非改 evolveOnce; evolveOnce 只能漂下一卷采样, 物理碰不到已落盘正文——这是合理的层次取舍, 反证缺一个"管单章"的环节) |

# 二、若需大修, 做哪个(ROI 排序)

**大概率结论已被三路一致命中: 章级自修订/编辑 pass(A)是最高杠杆**——一次同时治密度/啰嗦/节奏/章内重复多个, 补的正是"无编辑环节"这个结构缺失。它不是"第7遍 LLM", 是给生成补上唯一缺失的**全章回看视野层级**(prompt=局部视野、进化=跨章慢反馈, 结构上都给不了这个新信息源)。

## 🥇 A · 章级自修订/编辑 pass —— **ship(做这一个就回本)**

- **改哪**: `longrun.ts:214-216`(text 拼装完、`return` 之前)插入 `editChapter(text, goal)`: 给【全章草稿 + 独立 EDIT_CHECKLIST + 只删不增指令 + canonHard硬事实】跑一次 LLM(thinking:false, 低温~0.4 求稳不发散), 产收紧版替换 text。落盘在 `:435` 之前。
- **治**: H-1重描 / H-2拖沓 / H-3节奏 / M-1章内重复 —— **一招压4个**, 因为它补的是"编辑"这个缺失动作; 也接管了 evolveOnce 接不了的 L-1(单章质量)责任。
- **成本(token)**: 落在**最贵的同步主路**。R3/R2 校准: 编辑读全章~3600字+写~3000字 ≈ 1.5段正文 token → **token 维度 +18-22%**(非均匀+30%, 因 outline/title/反思都是短prompt); 但**延迟维度更糟**——整章级阻塞(`:429 await`), 每章 wall-clock 多一个长LLM往返, 4世界并发=4×吞吐损失。可降本: 只对 `breath<6 或 var异常` 的章触发(warm-fitness:149 已有 breath 信号当触发器), 非全章都修。
- **风险**: ⚠ 过度编辑→干瘪/丢温润留白/改崩一致性; ⚠ 修订后变短可能触发 `:431` floor(`<MINLEN*0.4`)退避重写整章 → 一次坏编辑=浪费1遍编辑+1整章。**必须带确定性三闸(见下), 无闸的 A 是 block**。
- **不打架**: ✅ 与进化正交(改初稿不改基因); ✅ 与 M1/M2 降密度同向(修订指令就是删冗破密度)。**但 2 条架构红线必守**(见四)。

**强制配套——确定性三道闸(M-1, 必须先于编辑逻辑落地)**: editChapter 是**可丢弃的纯增益层**, 任一闸不过→原样返回 draft, 编辑**永不致章变差**(最坏=现状, 下行风险钳到零, 这是 ROI 判断的基石):
1. **长度地板**: `edited.compact ≥ draft.compact*0.85`(删幅≤15%), 否则弃编辑用原草稿。顺带堵住"删太狠触发整章重写"的级联浪费。
2. **一致性地板**: 复用 `canonHard`(longrun deriveCanon 权威硬事实: 境界/派系/生死/恩怨)做廉价子串校验, 硬事实里角色名/境界词若编辑后凭空消失→弃编辑。比再调一次LLM省钱、复用既有权威源。
3. **留白地板(GENTLE)**: `|dlg1k(edited)-dlg1k(draft)| ≤ 2.0`, 防把寡淡留白删成情报播报或反向。直接复用 `warm-fitness.ts:149` 的 dlg/1k 算法(抽成共享util)。

**强制配套——切断 W_breath 双控源**: A 上线**同步把 `warm-fitness.ts:166` 的 W_breath 权重 0.08→0**(密度改由编辑pass确定性治理, 不再进化; 权重还给 var: 0.17→0.25)。否则: 编辑pass 推低 dlg/1k → W_breath 升 → 进化误以为"基因变好"朝那个方向爬山 → **把编辑的功劳错记到采样参数头上, 基因朝错误方向漂坏**(典型奖励错配)。这与 evolve.ts:141 弃用物象 obj 项同类决策: **能确定性做的别交给进化**。

## 🥈 D · 7块硬注入改"按需注入"(首末段全注、中段只注delta) —— **ship, 与A同期/先行**

- **改哪**: `longrun.ts:203` 的 loreBlock/canonInject/conBlock 这类一致性硬块改为**仅 `i===0 || last` 注入, 中间段省**(首段定调、末段收口)。
- **治**: H-1 的次要诱因——secPrompt 当前每段塞 sys+scene+ambience+crisis+ros+prev+beat+weave+PENMANSHIP+canonHard+loreBlock+canonInject+conBlock+evoGuidance **≈14块**, 逼每段都想交代所有注入项→均质化cram。A 在下游删冗补救不了上游cram, **D 是比 A 更便宜的一刀**(几乎零token, 纯prompt重排)。
- **成本**: 极低(纯prompt工程)。**风险**: ⚠ canonHard/conBlock 是一致性硬约束, 精简要保功能(改"首段全注、后续段只注delta")。
- **判定**: **先行或与A同期**, 先验证"降cram能否单独缓解一部分密度"。收益是候选里偏小, 别指望它单独解决密度。

## 🥉 B · 软化 perSec=900 填满约束 —— **ship with caveat(A 验证后再做)**

- **改哪**: `longrun.ts:197` perSec 改软上限/区间(**关键: 松"段配额"而非松"章下限"——perSec 才是元凶, MINLEN 反而相对无辜**); `:203` 段末"约900字"改"一段一件事写从容即收、几十到几百字皆可"; `:431` floor 保留(MINLEN 锚"疑似生成失败"是 load-bearing, 不能去)。
- **治**: H-1/H-2/H-3 的**结构源头**(不再逼填满, 直击C类根因)。比 A 更治本。
- **反直觉实测(R3, 必看)**: 章长 cv 已 0.16-0.20、范围 2883-7673 → **章长其实已在自然浮动, 900 不是硬钉死是软引力** → B 收益没"推手1"标题暗示的那么大, 不会让分布突变, 只把中央质量往下挪、方差放开。dlg/1k 偏高的**更大份额来自段内**(段cv0.46-0.66 >> 章cv0.16) → **正确打法是松段配额。**
- **成本**: 低(~10行常量+措辞)。**风险**: ⚠⚠ DeepSeek 去字数底后**可能直接写太短**(ch-30/176 已贴 floor 说明字数压力是当前长度的支撑物); ⚠ 影响 `evolve.ts:332` antiProxy 的 `m.len>avgLen*1.6` 长度判定(方差变大需重标定); ⚠ 撞"每章≥3000字"产品契约。
- **判定**: **必须 A 落地、确认修订pass能托住质量后再做 B**, 否则去字数底+无编辑兜底=塌。

## 4️⃣ C · 章级规划情绪弧(派不同perSec/采样给不同拍) —— **refactor first, 别新增独立pass**

- **改哪**: **扩展现有 `longrun.ts:178-186` beatSpec 让它附带"每拍详略/长度权重"**, 而非新加一个规划pass(否则抽象层+2, token+40%与A叠加)。让 perSec(:197)和采样(:206)按拍位浮动。
- **治**: H-2详略 / H-3节奏的**最干净解**(写前先规划哪拍详哪拍略)。但与 beatSpec 功能重叠。
- **判定**: **把详略权重并进现有 beatSpec**, 不要独立pass。优先级最低, A+B 落地后视效果再说。

# 三、增量 vs 大修 路线推荐(务实)

**推荐路线 = 先把 prompt 补丁打满+观测 → 不够再上"自修订pass"这个最小大修, 而非一上来推倒。** 这正是当前应走的路, 且现在**已走到"不够"的拐点**(一·铁证: 补丁全量生效 dlg/1k 仍偏2x)。

落地顺序(ROI 降序, 务实):
1. **D 先行/同期**(零token): `:203` 硬块改首末段注入, 先看降cram 能否单独缓解一部分密度。
2. **A 上线, 必带三道确定性闸 + W_breath归零**(这是"最小大修"的本体, 回本最快、压4病、下行钳零)。
3. **A 验证后再 B**(松段配额, 治本, 但必须有A兜底才安全)。
4. **C 并进 beatSpec**(给拍附详略权重), 不新增独立pass, 优先级最低。

## 「何时该上大修(A)」的触发条件(机读判据)

满足**任意一条**即该从增量切到 A:
- **[已满足] 密度天花板**: prompt/采样补丁全量 ship 后, live 世界 dlg/1k 中位数仍 > gold 上界(5.4)的 1.4x, 或隔章 cv > 0.4(单章抖动按不住)。← **现状 7.8-12.9 已远超, 触发**。
- **[观测中] 边际递减**: 连续 2 个 prompt 版本(如 M4→M5)的 dlg/1k 改善 < 0.5/章(补丁进入平台期)。
- **章内重复硬证**: 任一 live 章被人工/脚本抓到同一物象重描 ≥3 次(W_var 结构盲, 只能靠 A 抓)。
- **详略/节奏**: 章长 cv 持续 < 0.20 且段长 cv > 0.4(节奏被结构钉死, 段级注水), prompt 改不动。

**反向(暂不上大修, 继续增量)**: 若某条 prompt 补丁让 dlg/1k 中位数单调逼近 gold 且 cv 收窄 < 0.3, 说明还在 A 类可治区间, 继续堆补丁更省。

# 四、大修的风险与"不该动的"(别为大修而大修)

**保住能跑的、留白/温润质感、确定性resume、爽文隔离——这四条是 hard red line, 任何大修不得破坏。**

1. **保住能跑的(下行钳零)**: A 必须是**可丢弃纯增益层**——三道闸任一不过即回退原草稿, 最坏=现状。**M-1 的"编辑失败回退原草稿"必须先于编辑逻辑落地**; 守不住这条, A 就是给正文主路引入单点故障, 那种情况下应 block。
2. **留白/温润质感(别把克制删成播报)**: GENTLE 编辑 rubric 必须"删啰嗦但**保留白**", 留白地板(±2.0 dlg/1k)硬兜底。**编辑checklist 必须 `GENTLE?` 分叉**(与 PENMANSHIP/beatSpec 同款模式): GENTLE侧=「删过场/破静物循环/保留白/限对白爆」, 爽文侧=「砍意象过载/合并冗余角色/收紧视角」——**同一checklist治不了两种病**(GENTLE病=琐碎铺三段+静物循环; 爽文病=意象过载+名字生成失控)。
3. **确定性 resume**: A 在 writeChapter 内、落盘前完成, 不引入跨turn状态; 不碰 `:438-442` 的 pledger.turn 自增与 writtenBeats 滚动(T2进展账本是resume确定性的载体)。编辑是同章内同步动作, 天然 resume-safe。
4. **爽文隔离(零变更)**: GENTLE=false 世界必须逐字节同现状。A/D/B 的所有改动走 `GENTLE?` 门控或对爽文保持旧行为(B 的 MINLEN 软化只对 GENTLE)。这是已验证的向后兼容铁律(evolve.ts:45)。
