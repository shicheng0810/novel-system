# Backend Boundary Review — Vite-middleware HTTP Server

**Date:** 2026-05-10
**Scope:**
- `workbench/src/server.ts` (1692 lines, ~57 KB)
- `workbench/src/contracts.ts` (236 lines)
- `web/src/server/middleware.ts` (5 lines, re-export)
- `web/vite.config.ts` (27 lines, HMR injection)
- Cross-references: `src/parser.ts`, `src/ai-settings.ts`, `src/graph-raph-runtime-daemon.ts` (lifecycle)

**Threat model:** currently single-user, same-origin (Vite dev server → React).
W5 may expose to public internet via Cloudflare. Findings flag both present and future risks.

---

## Findings

### 1. Path traversal — atlas root resolved against caller-controlled segment, but symlinks bypass containment

```
SEVERITY: MEDIUM
workbench/src/server.ts:928-944  (StudioSession.atlasFile)
WHY: `lineAtlasRoot()` validates `lineId` against `^[A-Za-z0-9_-]+$` and rejects `..`, then `resolve(root, path)` is checked via `isInsideDirectory()` — solid against `..` traversal. BUT containment uses `path.relative()` only; if any file or directory inside `atlas/canon/` is a symlink pointing outside the root, the resolved real path is never checked. With `stat()` (not `lstat()`), Node follows the symlink and reads the target. Atlas content is generated by `AtlasCompiler`, but a malicious world-draft → compile path could plant a symlink-named file (e.g. through future arbitrary atlas filenames). Also `path` query parameter accepts absolute paths; `resolve(root, "/etc/passwd")` returns `/etc/passwd`, then `isInsideDirectory` correctly rejects it — but the absolute-path branch isn't documented and a future refactor of `isInsideDirectory` could regress.
FIX: Use `realpath` of both root and absolutePath, then re-test containment; explicitly reject `isAbsolute(path)` at request layer.
```

### 2. World-draft parse runs synchronously on the request thread with no length cap

```
SEVERITY: MEDIUM
workbench/src/server.ts:475, 519-521, 1582-1584  (apply-world flow → parseWorldDraft)
WHY: `POST /api/world/apply` reads body (unbounded — see #4), then `applyWorld` calls `parseWorldDraft(draftText)` synchronously inside `WorldHistoryEngine`. The parser splits by lines, then iterates; not catastrophic regex-DoS exposure, but `splitList` uses `/[，,、]/` and `parseAttributes` does string-only operations — safe from ReDoS. However: the engine then `runStage`s sample stages, which is CPU-bound and synchronous, blocking the event loop. A 50 MB draft will freeze every concurrent request for many seconds.
FIX: Reject draft bodies > 256 KB before parse; consider running heavy compile on a worker thread.
```

### 3. JSON body has no size limit — memory-exhaustion DoS

```
SEVERITY: HIGH (future) / MEDIUM (now)
workbench/src/server.ts:947-957  (readJsonBody)
WHY: `for await (const chunk of request)` accumulates all chunks into memory before `Buffer.concat` + `JSON.parse`. No `Content-Length` check, no streaming-size limit. A single client can stream gigabytes; Node will OOM the dev server. Localhost today, but if exposed via Cloudflare (W5), this is a cheap remote DoS. Vite's default middleware does not impose a body limit either.
FIX: Track running byte count in the loop and reject ≥ 1 MB (or whatever the largest legitimate world-draft requires).
```

### 4. JSON.parse error path leaks via 500 response — uncaught synchronous throw inside async

```
SEVERITY: LOW
workbench/src/server.ts:956, 1687-1690  (readJsonBody / catch)
WHY: When `JSON.parse` throws on malformed body, the outer try/catch returns 500 with `error.message` (e.g. "Unexpected token } in JSON at position 47"). This is the right status for malformed clients (should be 400) and slightly leaks parser internals. Not a security issue today, but promotes incorrect API contract. The catch is otherwise correct — async errors are caught.
FIX: Wrap `readJsonBody` in try/catch; return 400 `{error:"Invalid JSON body"}` on parse failure.
```

### 5. Error responses surface raw `error.message` from deep modules — paths and config leak

```
SEVERITY: MEDIUM
workbench/src/server.ts:1687-1690  (top-level catch)
WHY: `error instanceof Error ? error.message : ...` is sent to the client as JSON. Modules below the boundary (better-sqlite3, fs/promises, AtlasCompiler) emit messages containing absolute paths (`ENOENT: no such file or directory, open '/Users/chris0810/.../runtime-daemon.sqlite'`). On localhost this just shows in browser devtools, but on Cloudflare-exposed deploy this leaks the real filesystem layout, runtime-root path, and operator username. Stack traces aren't sent (good), but `.message` alone is informative.
FIX: Log the real error server-side; redact to a generic `{error:"Internal error", code:"E_GENERIC"}` (or a safe taxonomy) in the response.
```

### 6. Concurrent /api/runtime/start + /api/world/apply orphans the daemon and its SQLite checkpointer

```
SEVERITY: HIGH
workbench/src/server.ts:519-521, 465-489, 1257-1276, 1030-1086
WHY: `applyWorld → resetWithDraft` reassigns `this.engine = new WorldHistoryEngine(...)` and rebuilds memory/atlas stores. The `runtimeDaemon` (closure-scoped) was created with the OLD engine reference and an OLD `checkpointPath`. After reset, the daemon's `engine` and `agentRegistry`/`director` callbacks still hold references to the prior engine — but the kernel was constructed eagerly with the prior engine. There's no `daemon.stop()` or `daemon.dispose()` call on world-apply; new stage results sync to the new session's engine while the running daemon writes to a SQLite file that may now describe a discarded world. Also: rebuildStores `rm`s `memory/` and `atlas/` while the daemon is mid-tick → file races.
FIX: In `applyWorld`, await `runtimeDaemon.pause()` (and ideally a `stop()`/disposal), null out the closure ref, and reject the request if the daemon is `running`.
```

### 7. No mutex between simultaneous mutating requests on the single shared session

```
SEVERITY: MEDIUM
workbench/src/server.ts:975-1006, 530-548, 519-521, 674-681
WHY: `sessionPromise` resolves to ONE `StudioSession`. `runStage`, `applyWorld`, `promoteBranch`, `runAuto`, `confirmFinal` all mutate `this.engine`, `this.stageResults`, `this.currentDraft`, `this.lens`, etc. without any lock. `withSession` runs handlers concurrently if two requests arrive in flight; `runStageWithProvider` awaits a slow LLM call and then mutates. Two parallel `runStage` calls can produce interleaved stage results and corrupt `branchEvaluations`; `applyWorld` mid-`runAuto` rms the atlas dir under the auto-loop. Browser today only triggers one mutation at a time, but a misbehaving HMR reload, tab-double-click, or future API client breaks invariants silently.
FIX: Wrap mutating endpoints in a serialization queue (single-flight per session); read-only endpoints (status, tree, file) bypass.
```

### 8. `createUrl(input)` parses `request.url` without origin; query / pathname ambiguity if `req.url` is absolute

```
SEVERITY: LOW
workbench/src/server.ts:965-967, 1463-1464
WHY: `new URL(input, "http://localhost")` works for relative paths. RFC 7230 allows absolute-form request-target on proxies (`GET http://attacker.example/api/atlas/file?path=...`). Vite's connect middleware preserves whatever the client sends as `req.url`; in that absolute-form, the host overrides the base, but `parsedUrl.pathname` is still the path — generally safe. However, if request URL contains `\` (backslash) on some clients, `new URL` treats it differently from `path.join`. Atlas path lookup uses `parsedUrl.searchParams.get("path")` which is decoded — fine.
FIX: Reject `req.url` not starting with `/`; or validate `parsedUrl.host === "localhost"`.
```

### 9. AI-settings validate endpoint takes API key from request body, makes outbound HTTP — SSRF-adjacent

```
SEVERITY: MEDIUM
workbench/src/server.ts:1127-1149, 1561-1577  (validateAiSettings)
WHY: `POST /api/settings/ai/validate` accepts `baseUrl` from the request body and calls `validateDeepSeekConnection({baseUrl,...})`. There is NO allow-list of hosts. An attacker (or in localhost+Cloudflare-exposed mode, any unauthenticated caller) can point this at any internal HTTP endpoint (192.168.x.x, AWS metadata 169.254.169.254, internal services) with arbitrary headers. The response body is returned to the caller. Even if blind, a 200-vs-error timing oracle reveals reachability.
FIX: Restrict `baseUrl` host to a known allow-list (`api.deepseek.com`, optionally a configured proxy); strip 127.0.0.0/8, link-local, RFC1918 in production.
```

### 10. `clearAiSettings` does not propagate to the running runtime daemon

```
SEVERITY: MEDIUM
workbench/src/server.ts:1168-1178, 1059-1070
WHY: `DELETE /api/settings/ai` deletes the on-disk file but the daemon's `HttpAgentLLMProvider` was constructed in `getRuntimeDaemon()` with a captured `apiKey` and lives in a closure. After clearAiSettings, the daemon still ticks with the now-revoked key (which the user may have rotated specifically because of leak fear). `ensureConfigured()` blocks new requests but not the ALREADY-running daemon.
FIX: In `clearAiSettings`, await `runtimeDaemon?.pause()` and null out the daemon ref so next start rebuilds the LLM provider.
```

### 11. Runtime body parsing coerces with `Number()`; `NaN` reaches the daemon

```
SEVERITY: LOW
workbench/src/server.ts:1621-1631, 1607-1614
WHY: `POST /api/runtime/start` does `Number(body.targetTicks ?? 1)`. If client sends `"abc"`, `Number("abc") === NaN`. `RunAutoStagesRequestPayload.targetStageCount` similarly: `Math.max(1, NaN)` returns `NaN` (not 1) which then participates in `while (completedStages < NaN)` — false immediately, so loop runs zero times and the fallback path triggers; OK by accident here, but other consumers (`runtimeDaemon.start({targetTicks: NaN})`) may misbehave.
FIX: Use a `parsePositiveInt(value, default)` helper that rejects NaN; centralize in a runtime-input zod schema.
```

### 12. No runtime input validation — TypeScript types are dropped at the wire

```
SEVERITY: MEDIUM
workbench/src/server.ts:1542, 1577-1684
WHY: `body` is typed `Partial<WorkbenchRequest & RuntimeStartRequest>` but at runtime is `JSON.parse` of arbitrary bytes. `body.lens`, `body.draft`, `body.directive`, `body.qimenOverride`, `body.instructions`, `body.focusCharacterIds` flow into the engine, atlas compiler, and LLM prompts unvalidated. A malicious or bug-induced `focusCharacterIds: [{__proto__: {polluted:true}}]` → spread into a stage runner could pollute Object.prototype if any consumer uses `Object.assign({}, x)` then uses keys as truthy flags. Even without prototype pollution, `instructions` is passed to LLM prompts — the prompt-injection vector is real, and unvalidated nested types break the engine's assumptions.
FIX: Add zod (or hand-rolled) per-route validators between body-parse and handler call; strip unknown keys; use `Object.create(null)` or strict shape checks.
```

### 13. `/api/atlas/compile` and `/api/world/apply` perform unbounded filesystem writes without rate limit

```
SEVERITY: MEDIUM
workbench/src/server.ts:344-348, 519-521, 884-895, 908-917
WHY: Each `world/apply` does `rm -rf memory/` and `rm -rf atlas/`. Each `atlas/compile` writes the full atlas tree. Each `confirmFinal` writes per-scene memory + recompiles atlas. A loop of these is a disk-thrash + iops DoS, and tight loops can corrupt the SQLite checkpointer (open handle in another process). No cooldown, no per-IP limit (no IP awareness at all).
FIX: Per-session token bucket (e.g. 5 mutations / minute) and a serialization queue (which is also a fix for #7).
```

### 14. `/api/runs/:id` accepts arbitrary URL path segment as runId without validation

```
SEVERITY: LOW
workbench/src/server.ts:1497-1506, 1344-1355  (getRunDetail / runStore.loadRun)
WHY: `decodeURIComponent(parsedUrl.pathname.split("/").at(-1) ?? "")` is the runId; passed to `runStore.loadRun(runId)`. If `SimulationRunStore.loadRun` uses runId as a path component (it almost certainly does — runs are filed under `runtimeRoot/runs/<id>/`), and runId can contain `..%2F`, traversal is plausible. After `decodeURIComponent`, a value like `..%2F..%2Fetc` becomes `../../etc` — and that's what hits the store. Unverified here without reading SimulationRunStore, but this is exactly the wrapper layer that should enforce the constraint.
FIX: Validate runId against `^[A-Za-z0-9_-]{1,64}$` at the middleware layer (mirrors `assertSafeAtlasLineId`).
```

### 15. CORS not set + no CSRF token — relies on same-origin only

```
SEVERITY: MEDIUM (future)
workbench/src/server.ts:959-963 (sendJson never sets CORS / Vary)
WHY: No `Access-Control-Allow-Origin` is emitted, so cross-origin browser fetches are blocked by default — good for now. But a malicious page CAN send a `text/plain` simple POST to `http://localhost:5173/api/world/apply` (bypassing preflight because `Content-Type: text/plain` is a CORS-simple header), hitting `JSON.parse` of arbitrary text. Most routes will 500 silently, but `/api/settings/ai` (POST) and `/api/atlas/compile` (mutating side effects) will fire. Same-origin is the only protection. Once exposed via Cloudflare, this becomes a CSRF gold mine.
FIX: Reject POST without `Content-Type: application/json` (or a custom `X-Workbench: 1` header that no simple-CORS request can set); add CSRF token before any deployment.
```

### 16. State machine: `autoRunState.active` never cleared on error

```
SEVERITY: LOW
workbench/src/server.ts:609-665, 653-659
WHY: If `runStageWithProvider` throws inside the auto loop (LLM 500), the catch-all in middleware sends 500 but `this.autoRunState` is left at whatever the previous loop iteration set, with potentially `active: true`. The next session-state poll reports the daemon as still auto-running. Cosmetic but misleading; user UX shows "still running" until the next mutation.
FIX: Wrap the loop body in try/finally that updates `autoRunState` on both paths.
```

### 17. `runtimeStatus` route registered twice (GET + accidentally falls through to POST branch)

```
SEVERITY: LOW
workbench/src/server.ts:1493-1495, 1641-1643
WHY: The pre-POST switch at line 1493 handles `GET /api/runtime/status`. After the `if (request.method !== "POST") return 405` guard, line 1641 handles `/api/runtime/status` *again* — only reachable via POST. Both call the same handler. Not a bug per se (POST behaves same as GET for status), but indicates HTTP method confusion: you can MUTATE-OPERATION status with POST, which has empty body semantics — fine here, but suggests other endpoints may have similar duplication. Auditor should grep for repeated route paths.
FIX: Remove the duplicate POST-branch entry; explicitly enumerate which routes accept POST only.
```

### 18. MIME confusion: error responses always JSON, but no `X-Content-Type-Options: nosniff`

```
SEVERITY: LOW
workbench/src/server.ts:959-963
WHY: `sendJson` sets `Content-Type: application/json` correctly. But there's no `X-Content-Type-Options: nosniff`, no `Cache-Control`, no `Vary`. If atlas-file content (`AtlasFilePayload.content` is text — markdown) gets reflected to a future direct-serve route, MIME sniffing could cause IE/old-Chrome XSS. Plus error responses leak via cache if behind a CDN.
FIX: Set `X-Content-Type-Options: nosniff` and `Cache-Control: no-store` on every API response.
```
